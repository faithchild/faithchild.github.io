Redis内存存储之一、初识Redis

发表于 2018-08-14
字数统计: 2,418 | 阅读时长 ≈ 8

一、简介

  随着中国互联网的发展，网络在人们的生活中扮演着越来越重要的角色，方便着人们的日常生活包括出行旅游、支付等。当然在这个过程中也萌生了很多大型互联网公司比如阿里巴巴、携程、腾讯等，每个互联网公司的成功都离不开其背后的庞大的用户量，在这个用户量体系下关系型数据库可能能满足公司的需求，但是每个公司都遵从用户体验，关系型数据库虽然能满足业务的需求但是对于客户的体验感就会很差,比如浏览一个网页页面出来了但是白板一片的这种情况，这时候就出现了基于内存存储的非关系型数据库。比如Redis、memcache、MongoDB等。
  Redis是一个内存数据库，它不仅性能强劲，而且具备复制功能以及为解决复杂问题而存在的多种多样的数据结构。除了强大的数据结构以外，Redis可以通过复制、持久化、客户端分片等特性，利用多台Redis服务器集群来满足大型系统的需求。
  Redis即可以作为主存储库来保存业务中的数据，也可以作为关系型数据库的辅助存储库，多样性的使用使得Redis在大型系统的出场率大幅度的增加。当然至于Redis作为主存储库还是辅助存储库都是依赖于业务场景的，需要用户根据自己的业务场景选择。

二、数据库对比

  Redis这种非关系型数据库在小型公司或者业务量并不大的公司用的不多。通常公司会为了提高一点效率使用缓存而小型公司以及业务量一般的公司使用Memcache即可。大型公司公司也一般不会使用单个Redis服务器，通常会多个Redis服务器集群来为公司提供缓存服务。当然市面上有很多NoSql产品与关系型数据库产品我们主要了解下关系型数据库MySql与市面上常用的NoSql之间的差异:

名称 类型 存储的数据类型 查询方式 附加功能
Redis 内存存储非关系型数据库 字符串、列表、集合、散列表、有序集合5种 每种不同的数据类型都有自己的专属命令 发布与订阅、主从复制、持久化、脚本存储过程等
Memcache 内存存储键值对 键值之间的映射(key-value模式) 基本的CURD操作 缺陷:没有Redis的存储数据类型多、无法持久化数据
MySql 关系型数据库 很简单的库、表结构 Insert、Select、Delete、Update操作 主从复制、持久化、支持ACID特性
MongoDB 硬盘存储非关系型数据库 可以存储一些大文件 创建、读取、删除等基本操作 支持map-reduce操作、主从、分片等

不同的数据库解决不同的问题，比如Mysql等关系型数据库天生满足ACID特性比如简单模式下使用memcache可以替代复杂的Redis，当存储的数据是大文件时就可以使用MangoDB，具体的使用也是需要根据业务量来进行选择的。

三、Redis数据结构

  Redis中可以存储5种不同的数据，包括STRING(字符串)、LIST(列表)、SET(集合)、HASH(散列)和ZSET(有序集合)，下面展示了它们之间的区别:

结构类型 结构存储的值 读写操作的功能
STRING 可以是字符串、整数或者浮点数 对整个字符串或者字符串的其中一部分执行操作，对整数和浮点数执行自增或者自减的操作，保存的不单单是简单的字符串类型
LIST 一个链表，链表上的每个节点都包含了一个字符串 从链表的两端推入或者弹出元素，根据偏移量对链表进行修剪，读取单个或者多个元素，根据值查找或者移除
SET 包含字符串的无序收集器，并且包含的每个字符串都是独一无二、各不相同的 添加、获取、移除单个元素，检查一个元素是否存在与集合中，计算交集、并集、差集，从集合里面随机获取元素
HASH 包含键值对的无序散列表 添加、获取、移除单个键值对，获取所有键值对
ZSET 字符串成员与浮点数分值之间的有序映射，元素的排列顺序由分值的大小决定 添加、获取删除单个元素，根据分值范围或者成员来获取元素

虽然Redis中存储元素的数据结构有这五种，但是对于操这些数据结构的命令却很多。这也体现了Redis的强大之处，可以处理多种多样的数据，并且对多种多样的数据提供了多样的命令处理。

四、持久化

  我们知道关系型数据库中保存的数据是永久的除非去数据库中进行删除而内存数据库不同，内存数据库中的数据都是保存在机器内存中的，当机器出现问题或者突然断电等问题，存放在内存中的数据就完全消失了。这种情况是某些场景所无法容忍的，但是并不是没有解决方法，Redis提供了两种持久化方式:
  1. RDB持久化:通常RDB持久化被称为快照持久，它可以将某一时刻的所有数据都写入硬盘里面。
  2. AOF持久化:通常AOF持久化被称为只追加文件，它会在执行写命令时，将被执行的写命令复制到硬盘中。
两种持久化方式即可以一起使用也可以分开使用。RDB持久化可能会造成丢失最近生成快照之后写入的所有数据，如果生成快照的时间间隔长就可能丢失大部分的最新写入数据。AOF持久化是将每次执行的写命令保存到硬盘中，下次只要运行这个文件就能恢复Redis中的所有数据，相比RDB持久化AOF持久化可以大概率的减少丢失的数据，但是AOF也有缺陷就是会造成保存在硬盘上的AOF文件越来越大，从而导致机器的硬盘可存储量越来越少最终无法保存其它文件。当然对于不同的持久化问题有不同的解决办法这边就先不介绍了只是简单了解持久化的方式。不同的持久化方式也需要用户根据不同的场景选择不同的处理方式。

五、复制

  通常而言，公司里面使用Redis不会只使用一台服务器，因为一旦这台Redis发生问题时，那么Redis就无法为外界其它系统提供服务，这就会导致业务出现问题。所以一般都会有个两三台或者更多台组成一个集群给外界提供服务。这时候就问出现一个问题如何保证当我们向主服务器写入数据的时候，从服务器能够更新自己的数据获取最新的数据进行保存。以上问题的解决办法就是复制，通过主服务器与从服务器之间的复制功能来保证每次从服务器在主服务器写入新数据时就进行更新自己的数据。

六、事务

  我们知道关系型数据库支持事务，在关系型数据库中是这样来描述事务的，事务保证了一组SQL语句只有同时执行成功的时候才算成功才可以进行commit操作，只要一个语句出现问题就会进行rollback回滚的操作。
  Redis中的事务和传统关系型数据的事务有些区别，Redis中的事务是通过MULTI命令开始，然后可以执行批量命令最后以EXEC结束。这种事务处理存在一些缺陷，因为它不像数据库一样会报异常，因此无法直观的判断Redis命令是否执行正确。当然在这只是简单的介绍一下Redis事务。

七、集群

  在复制功能的时候说过，一般公司使用Redis不会使用一台因为这样的架构是没有容错性可言的，一旦Redis系统发生奔溃就发生崩溃了系统也就会导致奔溃。所以一般都是多台机器组合而成的集群，Redis中的集群通过分区(Cluster)方式实现，或者哨兵(Sentinel)解决。

八、总结

  Redis作为大型公司常用的缓存层架构的技术选型，有着它独有的优势和业界解决方案，多样性的数据结构、持久化、集群等都是其在大型系统中出场率提升的关键。
